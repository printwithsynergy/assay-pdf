# ADR-0001: Deterministic generation vs. committed corpus

**Status:** Accepted, v0.1.0.

**Context:** AssayPDF needs ~62 PDFs (v0.1.0) and ~175 PDFs (v0.1.1+) accessible in the repo for benchmarking. Three options:

1. **A: Commit the PDFs raw** — every clone gets the full corpus, no generation needed.
2. **B: Git LFS** — committed but stored in LFS, slower clones, requires git-lfs install.
3. **C: Deterministic generation** — commit the manifest only; PDFs are produced by `uv run assay generate`. Same code + same seed = byte-identical PDFs.

**Decision:** Option C — deterministic generation, manifest committed, PDFs gitignored.

## Reasoning

- **Repo size.** 62 PDFs at ~3KB average = 185KB now, but v0.1.1 boundary stress pushes toward 175 files at potentially 10–50 KB each = 2–8 MB of binary blobs. By v1.0 with proper image-resolution and TAC-stress generation it could be 100+ MB. Committed binary blobs compound forever in git history.
- **Forces engineering rigor.** Deterministic generation requires the generator to be byte-stable. That's good discipline anyway — if the generator isn't deterministic, the project's "reproducible accuracy" thesis cracks. Making determinism a build-time requirement means it gets tested on every CI run.
- **Manifest-as-source-of-truth.** With option C, `corpus/manifest.json` is the canonical record of what the corpus contains: file paths, expected outcomes, applicable variants, deterministic input parameters, and SHA-256 of the generated bytes. Reviewers can diff manifest changes cleanly even when the underlying PDFs drift.
- **30-second regen cost.** `assay generate` produces all 62 v0.1.0 PDFs in under 30 seconds on Intel. Cheaper than fetching 100 MB of LFS blobs over a slow connection.

## What we give up

- **Offline first-clone usage.** A new contributor can't immediately inspect a corpus PDF without running `uv run assay generate`. Mitigation: the manifest tells them exactly what each PDF contains and how to regenerate.
- **Bisecting binary regressions.** If a generator change accidentally produces a different PDF for the same rule, you can't `git diff` between corpus states. Mitigation: the manifest's SHA-256 acts as a tracer — if it changes unexpectedly, that's the diff.
- **External dependency on the generator.** If someone forks AssayPDF without intending to run the generator, they get an empty `corpus/`. Mitigation: README's quick-start makes `assay generate` the third command.

## Alternatives reconsidered

If the corpus grows past ~10 MB and the regen cost exceeds ~5 minutes, revisit option B (LFS). At v1.0 boundary stress with image-heavy negatives, this might be worth it. Until then, C is the right call.

## Implementation reference

See `src/assay_pdf/generator/determinism.py` for the metadata override layer and `tests/test_determinism.py` for the byte-equality test.
